1.1 布局：样式与其配置

1.1.1 app 的局限性和 app"内与外"
编写的小程序是在一个webview的框体内,所有开发局限在这个"页面内",而在外部,框架所搭建好的"页面内"的内容只能通过官方的API去操控.

打个比方,iphone6的整机像素是375px*667px,这里是逻辑像素.
"页面内"为一个像素大小为 375px*603px的webview,
"页面内"元素,我描述他为在视口元素外的一切,诸如顶部的状态栏,app顶部的导航栏,甚至是层叠于webview之上的默认tabbar.

在这里和一般的webapp,原生app局限就体现出来了, 当我们有对这些视口外的元素有样式,业务需求时,但官方接口没有相关支持,就会头疼.甚至更过分的是在这些外部元素出现bug时,我们也无法直接修改,只能通过一些其他手段去尝试弥补.

1.2.1 父到子
  • props (sender.通用 receiver.子组件) :
    • 优点:直接传递,父直接使用 setData 可以使子组件渲染
    • 缺点:仅限父子关系传递.
  • app.globalData (通用) :
    • 优点:广播性质, 持久化保存.app 不被消灭状态一直在.数据数量,类型不受限.
    • 缺点:不直接,需要父修改后子组件需要监控方法/或者生命钩子内触发调用.特别指出:组件没有 onShow 钩子,有这块需求做数据更新时非常不方便.
    • 优化方案:调用 globalData 数据时在页面单位进行操作后通过 props 传递.
      或者在子组件使用 getCurrentPage(0).methodName()操作
  • url 传值 (sender.通用,receiver.子页面)
    • 优点: 简单方便
    • 缺点:不适合大量内容传递,url 过长不易读,而且内容对象将转化为字符串.可能需要二次判断.
1.2.2 子到父
  • app.globalData(通用):
    • 同上
  • getCurrentPage(0)
    • getCurrentPage(0).data, 如果有渲染需求,请调用有 setData()的方法进行操作

1.3 钩子与异步

1.3.1 钩子和异步的配合问题
  • 代码
data:{isInitial:true}
  onLoad() {
    dd.getStorage({
      key: 'destroyed',
      success: res => { // 成功的回调
        if (res.data) {...} else {... } // 开关tabbar的入口
        this.pageSwitch(); // 根据入口显示页面方法;
        this.data.isInitial = false; // 第一次加载完毕,关掉Initial Flag
      },
      fail: res => { // 失败的回调
        console.log('获取失败', res);
      }
    });
  },
  onShow() {
    if (!this.data.isInitial) {
      this.pageSwitch(); // 根据入口显示页面方法
    } else {
      console.log('本次为首次渲染,onShow钩子已被阻止');
    }
  },

1.情景:app启动时,首页通过onLoad钩子访问本地缓存,根据缓存来决定tabbar各个入口是否显示, 并用pageSwitch方式来切换页面.
并且! 在从其他页回到首页时,onShow钩子同样调用pageSwitch方式来切换页面。
（如：首次加载，缓存表示页面入口1，2关闭，pageSwitch将会加载页面3，之后用户通过设置打开了页面入口1，再次返回主页，就会切换至页面1）.

onShow钩子自然是在onLoad完成后触发,但是一旦请求异步，请求的回调函数将排入执行队列末尾，在getStorage后马上执行onShow的pageSwitch,之后在执行请求的回调。这里的函数时序出了问题，最终页面会“闪一下”后进入想要的页面.

钉钉的确提供了同步获取缓存方法:getStorageSync(),但是安卓上不正常工作.

2.解决办法:
首先：data 设立一个 flag(isInitial);
其次：onLoad 在有回调动作时(比如拉取本地缓存),通过调整 flag 来控制 onShow 是否执行.

- 缺点:onLoad 内异步回调也得写一次.

3.bug: (不重要).调试器表现,getStorage 是同步的,所以下面的代码在手机正常,在模拟器上重复渲染一次(已验证:安卓,ios 表现一致)

2.1.3 a:for a:if 同时使用

在 vue v-for 的优先级高于 v-if,会导致列表渲染出的每一项都有 v-if
和 vue 不同,这两个没有明确表明不建议同时使用
但经过测试 a:for 优先级也是高于 a:if 的

<!-- for&if -->
<view a:for="{{forTest}}" a:if="{{item}}">
  {{item}}
</view>

// js
data:{
  forTest:[0,1,0,1],
  item:false
}


所以若出于性能考虑,请处理列表或是 a:if 作为外包,
对于列表内部渲染内容有 if 判断需求时,处理内容简单时可用

2.2.1 组件标签上的 class 将被覆盖

  • 页面上对组件标签使用样式会被组件样式底层覆盖掉

  • 组件挂载后,页面上的挂载点上的 component 标签将被销毁,像一个 slot 直接插入组件内内容.

<view>
  <test class="outer"></test>
</view>

<!-- test -->
<view>
  <view class="inner">
    Test Component
  </view>
</view>


不会存在一个带 class 为 outer 的标签

解决思路: 首先不要在组件标签直接使用 class, 在其外包一个 view 控制样式(这个方案也不完美), 另一个比较麻烦的解决方案是在 component 上开放接口 props,对要自定义的内容进行组件传值自定义.

2.3.1 节流函数写法
// 需求函数js:
import { throttle } from "/utils/util";
viewDetail: throttle(e => {
  console.log("触发", e);
}, 5000);

// utils/util.js
export function throttle(fn, wait) {
  if (!wait) wait = 1000;
  let end,
    begin = 0;
  return function() {
    end = Date.now();
    if (end - begin < wait) {
      return console.log("已触发节流");
    } else {
      fn.apply(this, arguments);
    }
    begin = end;
  };
}

onTap = viewDetail, 启动 throttle 函数,throttle 函数直接返回了以下函数,apply 使内构建的新函数绑定上在触发者(viewDetail)的 this 和 e(参数 arguments)

特别注意: viewDetail:throttle(function(e){}) 不要写箭头函数,会强行绑定上下文作用域,而 throttle 函数在 util.js 内部,就是 undefined
